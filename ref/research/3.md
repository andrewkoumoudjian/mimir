## Decision: **Mimir should be one transaction-intelligence platform with two lenses**

Your current structure is a good base, but the product code should live almost entirely inside `mimir/`. Treat `brim/` and `valsoft/` as **challenge-source folders**: raw data, briefs, policies, reference docs. The final product should not feel like “Brim app + Valsoft app.” It should feel like:

> **Mimir: the wisdom layer over business transactions.**
> One canonical transaction spine. One evidence engine. One case queue. Two sub-dashboards: **Spend Intelligence** and **Fraud Intelligence**.

That maps cleanly to the briefs. Brim requires natural-language spend analytics, policy compliance, pre-approvals, and automated expense reports.  Valsoft requires suspicious-transaction scoring, human-readable explanations, reviewer triage, README/PRD/implementation plan, and an updated flagged CSV.  Brim’s policy gives concrete rules you can digitize immediately, including manager pre-authorization and receipts over $50, restrictions on alcohol, tip limits, car/parking/gas receipts, and no personal use of corporate cards. 

The strongest structure is:

```text
Mimir
├── Command Center
├── Spend Intelligence         # Brim sub-dashboard
├── Fraud Intelligence         # Valsoft sub-dashboard
├── Unified Risk Cases         # shared by both
├── Compliance Draft Lab       # FINTRAC/FATCA/CRS local-only
├── Policy Studio
└── Audit Log
```

The key design move: **everything becomes a `RiskCase` with evidence**.

A split expense can be a Brim policy case. A cross-card device reuse can be a Valsoft fraud case. A high-risk case can become a FINTRAC-compatible draft. A transaction can therefore have spend context, policy context, fraud context, compliance context, and reviewer context in the same side drawer.

# 3. Fraud research conclusions from arXiv: what matters for Mimir

The fraud literature points toward four practical ideas you should implement, not just mention.

## 1. Fraud is relational, not only row-level

Recent work on financial fraud detection emphasizes graph-based models because fraud often appears through relationships among users, merchants, devices, IPs, accounts, and transactions. A 2024 review on GNNs for financial fraud detection highlights their strength in capturing complex relational patterns, and a 2025 HGNN paper constructs heterogeneous transaction graphs with users, merchants, and transactions, using attention and temporal decay. ([arXiv][2])

**Mimir implementation:** build a heterogeneous entity graph, but do not train a full GNN in the hackathon.

Use graph-derived features:

```text
device_card_degree
ip_card_degree
merchant_card_degree
device_tx_count_24h
ip_tx_count_24h
merchant_burst_score
connected_component_size
shared_device_with_high_risk_card
shared_ip_with_high_risk_card
```

This gives you graph intelligence without overbuilding.

## 2. Fraud is temporal

STA-GT, a spatial-temporal graph transformer for transaction fraud detection, explicitly models both graph structure and temporal dependencies. Dynamic heterogeneous graph work also focuses on capturing temporal and heterogeneous information in fraud settings. ([arXiv][3])

**Mimir implementation:** implement rolling-window features:

```text
card_tx_count_5m
card_tx_count_1h
same_merchant_count_1h
same_amount_nearby_count_24h
merchant_global_burst_1h
device_global_burst_1h
minutes_since_last_card_tx
split_purchase_cluster_score
```

This is especially important because Brim explicitly gives an example of split transactions to avoid approval thresholds. 

## 3. Fraud detection is imbalanced and label-scarce

Fraud datasets are usually highly imbalanced. Recent work on unsupervised transaction fraud detection notes that labels are scarce and that Isolation Forest remains popular because it is scalable and easy to deploy. Older credit-card fraud work also emphasizes concept drift and class imbalance as core operational problems. ([arXiv][4])

**Mimir implementation:** use an unsupervised scoring stack, not a supervised classifier that pretends to know the labels.

Recommended stack:

```text
Mimir Sentinel v0
├── robust per-card baseline score
├── graph-risk score
├── temporal/velocity score
├── policy-risk score
├── IsolationForest anomaly percentile
├── ECOD/COPOD anomaly percentile
└── reason-code compiler
```

## 4. Explanations matter as much as scores

XAI-focused fraud work combines strong predictive models with explanations such as SHAP to make model decisions more transparent. ([arXiv][5]) For this hackathon, explicit reason codes will be more valuable than complex SHAP plots because the judge needs to understand the case instantly.

**Mimir implementation:** every flagged transaction gets a reason list:

```json
[
  {
    "code": "DEVICE_SHARED_ACROSS_CARDS",
    "severity": "critical",
    "message": "The same device appears on 5 different cards."
  },
  {
    "code": "AMOUNT_OUTLIER_FOR_CARD",
    "severity": "high",
    "message": "Amount is 14.2x this card's median transaction."
  },
  {
    "code": "NEW_MERCHANT_FOR_CARD",
    "severity": "medium",
    "message": "First transaction at this merchant for this card."
  }
]
```

That is exactly what Valsoft wants: not just a score, but a reviewer-ready explanation.

# 5. Why not a full GNN or transformer in the hackathon build?

Mention the research in the demo, but implement the practical v0.

Graph transformers, TabFormer-style sequence models, and temporal GNNs are relevant for future Mimir versions. IBM’s TabFormer repository, for example, includes code and a synthetic credit-card transaction dataset with 24M records, which is useful as a reference for larger-scale sequence modeling. ([GitHub][6]) But your Valsoft dataset is 1,000 rows across 50 cards. 

So the correct engineering answer is:

> “We used the literature to design Mimir Sentinel’s feature architecture: graph, time, baselines, imbalance-aware ranking, and explainability. We did not pretend that a deep model trained on 1,000 rows would be better than a robust graph-aware anomaly model.”

That will sound much more mature to judges.

# 9. Compliance: keep it local-only and clearly bounded

For FINTRAC, build a **local draft validator**, not a submission feature. FINTRAC’s API documentation describes secure system-to-system report submission and provides schemas, validation-message schemas, Swagger endpoints, and report-specific schemas such as Suspicious Transaction Report schemas. ([FINTRAC][7]) FINTRAC’s guidance page also makes clear that it publishes obligations and guidance for reporting entities, but that the guidance is not legal advice. ([FINTRAC][8])

Your implementation should say:

```text
FINTRAC-compatible draft generated.
Validated locally against saved schema.
Submission disabled.
No API keys.
No outbound request.
Human compliance review required.
```

For FATCA/CRS, use it as a **readiness simulator**, not a real filing system. CRA says Part XVIII and Part XIX returns are filed electronically through XML or Web Forms, and recommends validating XML against schema files before submission. ([Canada][9]) CRA also provides XML specifications for Part XVIII and Part XIX information returns. ([Canada][10]) HMRC’s CRS/FATCA repositories are useful as architecture inspiration for registration, FI management, reporting frontend, and upload flows, but they are UK systems and should not be treated as Canadian legal authority. ([GitHub][11])

For the demo, compliance should be a side panel:

```text
Compliance Draft Lab
├── FINTRAC STR-style JSON draft
├── local schema validation
├── missing fields
├── narrative generated from evidence
├── FATCA/CRS profile completeness
└── audit export
```

This is enough to look sophisticated without overclaiming.

# 12. Inference best practices

## Use batch mode for Valsoft

Valsoft gives you one month of 1,000 transactions. For this challenge, batch scoring is acceptable and probably expected. The hidden answer key evaluates the final flagged set, so use the full dataset to compute cross-card graph features.

But document this clearly:

```text
Mimir Sentinel v0 runs in offline review mode for the hackathon dataset.
In production, temporal features would be computed only from transactions available at decision time.
```

That protects you from “future leakage” criticism.

## Optimize for F1, not accuracy

Valsoft explicitly says it scores precision, recall, and F1, and warns against over-flagging.  This aligns with fraud research: in imbalanced fraud detection, precision/recall-oriented metrics are more meaningful than raw accuracy or ROC-AUC alone because false positives directly waste review effort. ([arXiv][12])

Your thresholding strategy should be:

```text
1. Score every transaction.
2. Sort descending.
3. Use expected fraud prevalence as a guide, not a hard rule.
4. Start with 7–10% flagged.
5. Manually inspect top clusters.
6. Adjust threshold to preserve precision.
7. Export final scored CSV.
```

## Freeze evidence at scoring time

When a case is created, store:

```text
case_snapshot_json
reason_codes_json
model_version
feature_values_json
```

Do not recompute reasons silently after reviewer actions. A real audit trail needs to show what the system believed at the time of decision.

## Build a session feedback loop

When a reviewer dismisses a case:

```text
- add reviewer action to audit log
- mark similar cases as “possibly lower confidence”
- visually show “3 related cases may be affected”
- optionally reduce score by a small session-only adjustment
```

Valsoft explicitly rewards a feedback loop that learns from reviewer decisions inside the session.
