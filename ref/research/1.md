# Best platform concept

## **RiskCanvas: Agentic Spend, Fraud, and Compliance Intelligence**

RiskCanvas has five visible modules:

1. **Unified Transaction Lake**
   Ingests both Brim and Valsoft CSVs, normalizes them into one canonical transaction schema, stores them as Parquet, and queries them through DuckDB/Polars.

2. **Agentic Visualization Canvas**
   User asks: “Show me risky cross-border spend by merchant category and explain the top cases.”
   The agent does **not** write arbitrary frontend code. It emits a validated JSON `VizSpec`. The dashboard renders that spec into KPI cards, tables, Vega-Lite charts, maps/heatmaps, timelines, and case views. Vega-Lite is particularly appropriate because it is a declarative JSON grammar for interactive graphics. ([Vega][15])

3. **Fraud/Risk Engine**
   Produces a `risk_score`, `risk_tier`, `rule_flags`, and `model_explanation` per transaction using:

   * deterministic rules,
   * unsupervised anomaly detection,
   * optional supervised/weakly supervised XGBoost,
   * SHAP explanations,
   * optional graph features.

4. **Compliance Dry-Run Lab**
   Builds **candidate** FINTRAC-style reports and FATCA/CRS due-diligence checklists. It validates payloads locally but never submits them. FINTRAC’s API documentation says the API supports system-to-system report transfers for report types including suspicious transaction, electronic funds transfer, large cash, large virtual currency, and casino disbursement reports, with single and bulk endpoint groups; your product should imitate validation/packaging only, with no credentials and no POST submission. ([FINTRAC][16])

5. **Human Investigation Queue**
   The platform never says “this is fraud” as an autonomous final decision. It says: “High-risk case, reasons, evidence, recommended next action.” A human can approve, reject, investigate, export evidence, or mark false positive. That ethical boundary matters because FINTRAC guidance also frames reporting and compliance around documented programs, KYC, monitoring, reporting, and record-keeping obligations—not autonomous AI decisioning. ([FINTRAC][17])

# The fraud/risk model you should build

Do not overbuild. For a hackathon, the best risk engine is a **hybrid rules + anomaly model**, not a giant ML system.

## Baseline risk rules

Implement deterministic rules first because they are explainable and demo well:

```text
RULE_CROSS_BORDER:
  cardholder_country != merchant_country

RULE_HIGH_AMOUNT:
  amount > p95(amount by card_id or merchant_category)

RULE_VELOCITY:
  count(transactions by card_id in 1h) > threshold

RULE_DEVICE_REUSE:
  same device_id used by many card_id values

RULE_IP_REUSE:
  same ip_address used by many card_id values

RULE_NEW_MERCHANT_FOR_CARD:
  merchant unseen for card_id historically

RULE_UNUSUAL_CHANNEL:
  channel not common for card_id or merchant_category

RULE_MERCHANT_CATEGORY_SPIKE:
  category spend materially above baseline
```

These rules work with your exact Valsoft fields and partially with Brim fields.

## ML layer

Start with **IsolationForest** for unsupervised anomaly detection because your provided Valsoft schema does not include a fraud label. Scikit-learn’s IsolationForest is designed to isolate anomalies through random feature splits, where anomalies tend to have shorter path lengths. ([Scikit-learn][18])

Add XGBoost only if you have labels or create weak labels from rules. XGBoost’s Python package supports standard Python model interfaces and is easy to run in a FastAPI/Polars pipeline. ([XGBoost Documentation][19]) Use SHAP for explanations if using tree models; SHAP’s TreeExplainer supports XGBoost, LightGBM, CatBoost, PySpark, and many tree-based scikit-learn models. ([SHAP][20])

## Feature set

Create these features in Polars:

```text

```

The mistake would be letting the agent produce arbitrary React, arbitrary SQL, or arbitrary chart code. The winning implementation is:

# Compliance implementation: impressive but safe

## FINTRAC: build a **no-send dry-run validator**

Your compliance module should do four things:

1. Detect whether a transaction/case is a **candidate suspicious activity case**.
2. Generate a local `fintrac_str_candidate.json`.
3. Validate it against your local schema/model.
4. Produce `validation_report.json` and `evidence_manifest.json`.

Do **not** implement real submission. The code should have a hard guard:

```python
def submit_to_fintrac(*args, **kwargs):
    raise RuntimeError("Submission disabled: hackathon dry-run validator only.")
```

FINTRAC’s API page says the report submission API supports secure system-to-system transfer, includes single and bulk endpoint groups, and provides report schemas, validation message schemas, Swagger docs, and sample reports. ([FINTRAC][16]) For your demo, use that as the model: “we are API-schema compatible and validation-ready, but this prototype never sends anything.”

Compliance output folder:

```text
case_TX123/
  evidence_manifest.json
  risk_explanation.json
  fintrac_str_candidate.json
  fintrac_validation_report.json
  fatca_crs_indicia_check.json
  audit_log.jsonl
```

## FATCA/CRS: do not pretend transaction data is enough

With only the fields you listed, you cannot truly determine FATCA/CRS reportability. You are missing account holder name, address, tax residence, TIN, citizenship/US indicia, entity classification, controlling persons, account number, account balance/value, and self-certification records.

So the correct product is a **FATCA/CRS due-diligence gap engine**, not a fake filing engine.

CRA’s CRS guidance says Canadian financial institutions have due diligence and reporting obligations under Part XIX and that account holders can be asked to certify or clarify tax residence status. ([Canada][22]) CRA’s Part XVIII/FATCA instructions require Canadian financial institutions to identify U.S. reportable accounts and annually report required information, using guidance, XML specifications, and schemas. ([Canada][23]) CRA’s Part XIX/CRS filing instructions also reference XML specifications and schema-based electronic filing. ([Canada][24]) The OECD resource centre provides CRS standards, implementation handbooks, consolidated CRS text, XML schema user guides, and schema ZIPs. ([OECD][25])

Your FATCA/CRS module should therefore output:

```json
{
  "case_id": "TX123",
  "fatca_crs_status": "insufficient_account_holder_data",
  "detected_indicia": [
    "cross_border_activity",
    "merchant_country_differs_from_cardholder_country"
  ],
  "missing_required_fields": [
    "account_holder_tax_residence",
    "TIN",
    "account_holder_address",
    "entity_classification",
    "controlling_persons"
  ],
  "recommended_next_step": "request_self_certification_or_account_profile_review"
}
```

That will sound much more credible than pretending you can file CRS/FATCA from transaction rows.

# Best technical stack for you and for Codex

## Recommended stack

Use **Python + Polars + FastAPI + Pydantic + DuckDB + React/TypeScript**.

This is the best balance of speed, credibility, and Codex-friendliness.

| Layer                 | Recommended tool                     | Why                                                                                                                                |
| --------------------- | ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| Package/runtime       | `uv`, Python 3.12                    | Fast setup, reproducible, Codex-friendly                                                                                           |
| DataFrames            | `polars`                             | Rust core, very fast, lazy queries, streaming/out-of-core, schema discipline. ([Polars User Guide][26])                            |
| Local analytics DB    | `duckdb`                             | Excellent for Parquet, efficient reads/writes, projection/filter pushdown. ([DuckDB][27])                                          |
| API                   | `fastapi`                            | Modern Python API framework based on type hints and OpenAPI/JSON Schema. ([FastAPI][28])                                           |
| Schemas               | `pydantic` v2                        | Generates JSON Schema from models; perfect for `VizSpec`, compliance payloads, and canonical transaction schemas. ([Pydantic][29]) |
| ML baseline           | `scikit-learn`                       | IsolationForest baseline                                                                                                           |
| ML advanced           | `xgboost`, `shap`                    | Optional supervised scoring + explainability                                                                                       |
| Frontend              | React + TypeScript + Vite or Next.js | Best for polished demo                                                                                                             |
| Charts                | Vega-Lite first, ECharts optional    | JSON-native chart specs                                                                                                            |
| Tables                | TanStack Table                       | Fast case queue                                                                                                                    |
| UI                    | shadcn/ui + Tailwind                 | Professional without design overhead                                                                                               |
| Compliance validation | `jsonschema`, `lxml`, `xmlschema`    | JSON/XML validation                                                                                                                |
| Testing               | `pytest`, `ruff`, `mypy` optional    | Codex guardrails                                                                                                                   |

# Codex implementation order

Give Codex this build sequence.

## Phase 1 — schemas and ingestion

Acceptance criteria:

```text
- Brim CSV loads with exact pasted columns.
- Valsoft CSV loads with exact pasted columns.
- Both normalize into UnifiedTransaction.
- Output writes to data/silver/unified_transactions.parquet.
- Tests fail if any required source column is missing.
```

## Phase 2 — risk features and rules

Acceptance criteria:

```text
- Feature table created from unified transactions.
- Rules produce explainable flags.
- Every high-risk case has rule_ids + human-readable reasons.
- Case queue endpoint returns JSON.
```

## Phase 3 — ML scoring

Acceptance criteria:

```text
- IsolationForest trains on numeric/categorical encoded features.
- Each transaction receives model_score.
- Final risk_score combines rules + model_score.
- Model metadata is saved with timestamp and feature list.
```

## Phase 4 — visualization agent

Acceptance criteria:

```text
- Agent receives a natural-language question.
- Agent returns only valid VizSpec JSON.
- Invalid specs are rejected with useful validation errors.
- Frontend renders KPI card, bar chart, table, and compliance preview.
```

## Phase 5 — compliance dry-run

Acceptance criteria:

```text
- A case can generate a FINTRAC-style STR candidate JSON.
- Local JSON Schema validation runs.
- No network submission exists.
- FATCA/CRS module reports missing account-holder fields instead of faking compliance.
```

## Phase 6 — demo polish

Acceptance criteria:

```text
- One-click sample data load.
- Three sample questions work.
- One high-risk case produces an evidence pack.
- UI has a polished “auditor/investigator” feel.
```
